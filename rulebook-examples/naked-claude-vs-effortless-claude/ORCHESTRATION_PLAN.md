# Orchestration Plan — `orchestrate.sh`

## Goal

A single entry-point script, `orchestrate.sh`, that runs one cell of the
**Tale of Two Claudes** experiment matrix end-to-end:

1. **Sets up** an isolated, git-tracked experiment folder for one of six
   combinations: `{Naked, Effortless} × {V1, V2, V3}`.
2. **Runs** the Claude Code CLI inside that folder with a fixed build prompt
   and watches it work.
3. **Samples token usage in the background** (via `ccusage` / `cc-usage`)
   every 30s, writing to a JSON log inside the experiment folder.
4. **Reports**: a separate script turns the JSON log into an HTML report
   showing token-usage-over-time for that run, with the option to compare
   across runs.

The goal is that any of the six experiments can be launched with one command,
and every run leaves behind enough artifacts (code, git history, token log,
HTML report) to compare apples-to-apples afterwards.

---

## File layout

All new files live at the repo root unless noted:

```
orchestrate.sh                  # interactive entry point (asks version + flavor)
tale-orchestrator.mjs           # the actual scaffolding logic (Node)
tale-token-sampler.mjs          # background ccusage poller (Node)
tale-report.mjs                 # JSON log → HTML report generator
Naked-Claude-Experiments/       # parent dir for all runs (created on first run)
  2026-04-06-1255-NakedV1/
    v1_SPECIFICATION.md
    CLAUDE.md                   # copied from claude_naked.md or claude_effortless.md
    .git/
    .tale/
      run.json                  # run metadata (flavor, version, start, end, exit)
      tokens.jsonl              # one sample per 30s from ccusage
      tokens.log                # raw ccusage stdout for debugging
      report.html               # generated by tale-report.mjs
```

`.tale/` is a sidecar dir kept *inside* the experiment folder so that the run
artifacts travel with the code, but it's `.gitignore`d in the experiment's own
git history (so the diff stays focused on what Claude built).

## `orchestrate.sh` — interactive entry point

A thin Bash wrapper. Its only job is to ask two questions and then
hand off to `tale-orchestrator.mjs`. Reasoning for keeping the interactive
shell separate from the Node logic: the Node script stays scriptable for a
future "run all six" loop, while `orchestrate.sh` is what a human types.

Behavior:

1. Prompt: `Which version? [1/2/3]` — validates input.
2. Prompt: `Which Claude? [n=naked / e=effortless]` — validates input.
3. Echoes back the chosen combination and asks for confirmation
   (`Run NakedV2? [y/N]`) — cheap guard against fat-fingering.
4. `exec node tale-orchestrator.mjs <flavor> <version>` — replaces the
   shell process so signals propagate cleanly.

No flags, no positional args. If you want non-interactive, call
`tale-orchestrator.mjs` directly.

## `tale-orchestrator.mjs` — the workhorse

This is the script described in the previous planning round, extended to
manage the background sampler. Pure Node, no extra deps.

### CLI

```
node tale-orchestrator.mjs <flavor> <version>
   flavor:  naked | effortless
   version: v1 | v2 | v3
```

Optional flags:
- `--dry-run` — print actions, do nothing.
- `--no-claude` — scaffold + start sampler, but skip the Claude CLI invocation.
- `--no-sampler` — scaffold + run Claude, but skip token sampling.

### Step-by-step

1. **Validate args.** Map to a label: `NakedV1`, `EffortlessV3`, etc.

2. **Compute the experiment folder path.**
   - Timestamp: `YYYY-MM-DD-HHMM` (local time, matches the
     `2026-04-06-1255-NakedV1` example).
   - Full path:
     `<repo-root>/Naked-Claude-Experiments/<timestamp>-<label>/`.
   - Abort if it already exists.

3. **Create dirs.** `Naked-Claude-Experiments/` (if missing), the experiment
   dir, and `<exp>/.tale/`.

4. **Copy files into the experiment dir:**
   - `<version>_SPECIFICATION.md` → same filename in dest.
   - `claude_<flavor>.md` → `<dest>/CLAUDE.md` (renamed on copy).
   - Write a minimal `.gitignore` to the experiment dir containing `.tale/`
     and `node_modules/`.

5. **Write `.tale/run.json`** with:
   ```json
   {
     "flavor": "naked",
     "version": "v1",
     "label": "NakedV1",
     "started_at": "2026-04-06T12:55:00-07:00",
     "experiment_dir": "...",
     "prompt": "..."
   }
   ```

6. **`git init` the experiment dir** and commit the scaffold
   (`scaffold: NakedV1 from v1 spec`). Use `execFileSync` with `cwd` set;
   never `cd` the parent process.

7. **Start the background token sampler** as a detached child:
   - `node tale-token-sampler.mjs --dir <exp> --interval 30`
   - stdout/stderr → `<exp>/.tale/tokens.log`
   - Record the PID into `<exp>/.tale/sampler.pid` so it can be cleaned up.
   - Use `child_process.spawn` with `detached: true, stdio: ['ignore', fd, fd]`
     and `unref()` so the orchestrator can wait on Claude without holding the
     sampler open.

8. **Invoke the Claude Code CLI** in the experiment dir:
   - `cwd` = experiment dir, `stdio: 'inherit'` so the user sees the session.
   - Build the prompt by reading `effortless.json` from the repo root,
     looking up `<version>-base-id` in `ProjectSettings`, and substituting
     both the spec filename and the base id into the template:
     > The `<version>_SPECIFICATION.md` describes the airtable `<base-id>`.
     > please write this as a node/vite-react app. There is a local
     > instance of postgres, with a db called effortless-demo.
   - Concrete example for `naked v2`:
     > The v2_SPECIFICATION.md describes the airtable app7G5emeY7miM4WN.
     > please write this as a node/vite-react app. There is a local instance
     > of postgres, with a db called effortless-demo.
   - Invoke as `claude -p "<prompt>"` (non-interactive print mode).

9. **Wait for Claude to exit.** Capture exit code.

10. **Stop the sampler.** Read `sampler.pid`, send `SIGTERM`, wait up to 5s,
    `SIGKILL` if needed. Delete the pidfile.

11. **Finalize `run.json`** with `ended_at`, `exit_code`, and a count of
    sampler entries written.

12. **Generate the HTML report** by invoking
    `node tale-report.mjs --dir <exp>` → writes `<exp>/.tale/report.html`.
    Print the absolute path to stdout so the user can open it immediately.

13. **Exit** with Claude's exit code (so a future "run all six" loop can
    detect failures).

### Signal handling
- On `SIGINT`/`SIGTERM` to the orchestrator: kill the sampler, finalize
  `run.json` with `exit_code: -1` and `interrupted: true`, still attempt to
  generate a partial report, then exit non-zero.

## `tale-token-sampler.mjs` — background poller

A small Node script with one job: every N seconds, ask `ccusage` what the
current token totals are, append a JSON line to `tokens.jsonl`, and loop
until killed.

### CLI

```
node tale-token-sampler.mjs --dir <experiment-dir> --interval <seconds>
```

### Behavior

1. Resolve the log path: `<dir>/.tale/tokens.jsonl`.
2. Loop forever:
   - `t0 = Date.now()`
   - Run `cc-usage --json` with `cwd` = the experiment dir. (`cc-usage` is
     project-scoped — running it inside the experiment dir gives totals for
     just that experiment, no global-baseline subtraction needed.) Capture
     stdout.
   - Parse the JSON. Store the entire `total` block (sessions, messages,
     tool_calls, tokens.{input,output,cache_read,cache_creation,total},
     api_equivalent_value_usd) plus a wall-clock timestamp and
     elapsed-seconds-since-start counter.
   - Append one line:
     ```json
     {"t":"2026-04-06T12:55:30-07:00","elapsed_s":30,"total":{...}}
     ```
   - Sleep until `t0 + interval*1000` (so the cadence stays steady even if
     cc-usage takes a few seconds).
3. On `SIGTERM`: flush, close the file, exit 0.
4. On `cc-usage` errors (non-zero exit, unparseable JSON, no sessions yet
   on disk): log an error line to `tokens.log` but keep the loop alive — a
   transient hiccup or an early sample taken before Claude has flushed any
   sessions should not lose the rest of the run.

### Why JSONL not JSON

Append-only, crash-safe, trivially parseable line by line. A power loss
mid-run leaves the previous samples intact. Easier for the report generator
to stream.

### Why a separate process, not a setInterval inside the orchestrator

- Survives if the orchestrator crashes (we still want the partial token log).
- Cleanly killable via PID without unwinding the orchestrator's event loop.
- Trivially reusable: you can attach the sampler to *any* directory after the
  fact, not just an orchestrated run.

### Scoping note
`cc-usage` is project-scoped (the wrapper Bash script invokes
`cc_usage.py` which keys off the cwd's project path). Because the experiment
dir is a fresh, isolated path, every sample's `total` is **already a
within-run cumulative** — no baseline subtraction needed. The report
computes per-30s deltas by subtracting consecutive samples.

## `tale-report.mjs` — JSON → HTML report

A standalone script that turns one run's `tokens.jsonl` into a self-contained
HTML file. No server, no build step — open it in a browser.

### CLI

```
node tale-report.mjs --dir <experiment-dir>
   # writes <dir>/.tale/report.html

node tale-report.mjs --compare <dir1> <dir2> [<dir3> ...]
   # writes a combined report at ./tale-comparison-<timestamp>.html
```

### Single-run report contents

1. **Header**: label (`NakedV1`), flavor, version, start/end timestamps,
   total wall-clock duration, Claude exit code, interrupted flag.
2. **Headline numbers**: total input tokens, total output tokens, total cost
   (if ccusage provides it), peak rate, samples taken.
3. **Time-series chart**: tokens used per 30-second window, on a single
   inline `<svg>` (no external chart library — keep the HTML self-contained
   and offline-viewable). X axis: elapsed seconds. Y axis: tokens added in
   that interval (delta from previous sample). Stacked or two-series for
   input vs output.
4. **Cumulative chart**: a second SVG with the running total over time.
5. **Sample table**: collapsible `<details>` containing every row of the
   JSONL as a sortable table — for spot-checking.
6. **Run metadata**: dump of `run.json` at the bottom.

### Comparison report contents

When invoked with `--compare`, overlay multiple runs on the same axes
(elapsed seconds on X, cumulative tokens on Y). One line per run, color-coded
by flavor (naked = one color, effortless = another), line style by version
(v1 solid, v2 dashed, v3 dotted). This is the report that actually tells the
story of the experiment — it's where you see the Naked curve diverge from
the Effortless curve as the spec gets bigger.

### Why hand-rolled SVG and not Chart.js / d3

- Zero dependencies, zero install step.
- The HTML file is portable — email it, drop it in a slide, archive it.
- The data volumes are tiny (a 10-minute run at 30s cadence is 20 points);
  hand-rolled SVG is more than enough.

## End-to-end flow, in one picture

```
user runs: ./orchestrate.sh
  ├─ asks version (1/2/3) and flavor (n/e), confirms
  └─ exec node tale-orchestrator.mjs <flavor> v<n>
       ├─ mkdir Naked-Claude-Experiments/<ts>-<label>/
       ├─ copy v<n>_SPECIFICATION.md
       ├─ copy claude_<flavor>.md → CLAUDE.md
       ├─ write .gitignore, run.json
       ├─ git init + initial commit
       ├─ spawn (detached) tale-token-sampler.mjs ──┐
       │                                            │ every 30s:
       │                                            │   run ccusage
       │                                            │   append to tokens.jsonl
       ├─ spawn claude CLI with fixed prompt        │
       │   (stdio inherited; user watches it work)  │
       ├─ wait for claude to exit                   │
       ├─ SIGTERM the sampler ─────────────────────┘
       ├─ finalize run.json
       ├─ node tale-report.mjs --dir <exp>
       │   └─ writes report.html
       └─ print report path, exit with claude's code
```

## What this plan deliberately defers

- **No "run all six" loop yet.** Once `orchestrate.sh` works for one cell,
  adding `orchestrate-all.sh` is a five-line wrapper. Don't build the loop
  before the cell works.
- **No automated grading of the resulting apps.** Whether the generated
  app actually passes its acceptance tests is a separate concern (a
  `tale-judge.mjs` for later). This plan only measures *token cost*, not
  *quality*.
- **No Postgres provisioning.** The fixed prompt assumes `effortless-demo`
  already exists locally. If a run needs a clean DB, that's a manual step
  outside the orchestrator.
- **No cross-machine portability.** The sampler shells out to whatever
  `ccusage` is on `PATH`. If it's missing, the sampler logs an error and
  the run still produces code + a (mostly empty) report.

## Resolved decisions

1. **Prompt templating** — **Yes, template the spec filename per version.**
   The prompt sent to Claude uses the actual file (`v1_SPECIFICATION.md`,
   `v2_SPECIFICATION.md`, or `v3_SPECIFICATION.md`) for the chosen run. Only
   one of the six combinations runs at a time; multi-run orchestration is a
   future shell wrapper, not this script's job.

2. **Base id substitution** — **Yes, substitute the real base id.** Pulled
   from `effortless.json` → `ProjectSettings`:
   - `v1` → `appgjoEcFNxluhbvK` (`v1-base-id`)
   - `v2` → `app7G5emeY7miM4WN` (`v2-base-id`)
   - `v3` → `appKLygCIXweUUKtM` (`v3-base-id`)

   The orchestrator reads `effortless.json` at startup and looks up
   `<version>-base-id` from `ProjectSettings`. The `app123xyz321` placeholder
   in the prompt template is replaced with that value.

   **Final prompt template** sent to Claude:
   > The `<version>_SPECIFICATION.md` describes the airtable `<base-id>`.
   > please write this as a node/vite-react app. There is a local instance
   > of postgres, with a db called effortless-demo.

3. **Claude CLI invocation** — **Non-interactive print mode.** Use
   `claude -p "<prompt>"` with `cwd` set to the experiment dir and `stdio`
   inherited so the user can watch it stream. (`claude` was not on PATH in
   the orchestration sandbox during planning — confirm the exact flag with
   `claude --help` on the user's shell before first run; `-p` is the
   documented non-interactive flag.)

4. **`cc-usage` invocation** — confirmed against the installed script at
   `/Users/eejai42/.local/bin/cc-usage` (a Bash wrapper around
   `cc_usage.py`). The sampler will run:

   ```
   cc-usage --json
   ```

   with `cwd` = the experiment directory. `cc-usage` is **project-scoped**
   (it keys off the cwd's project path), so running it from inside the
   experiment dir gives totals for **just that experiment** — no need to
   record deltas against a global baseline.

   The JSON output shape (confirmed by running it):

   ```json
   {
     "project": "<absolute path>",
     "branches": {
       "main": { "sessions": N, "messages": N, "tokens": N,
                 "api_equivalent_value_usd": N }
     },
     "total": {
       "sessions": N,
       "messages": N,
       "tool_calls": N,
       "tokens": {
         "input": N,
         "output": N,
         "cache_read": N,
         "cache_creation": N,
         "total": N
       },
       "api_equivalent_value_usd": N
     }
   }
   ```

   The sampler stores the entire `total` object plus a wall-clock timestamp
   and elapsed seconds. The report computes per-30s deltas from the
   cumulative `tokens.total` (and per-bucket deltas for input / output /
   cache_read / cache_creation).

   Caveat: `cc-usage` reads from Claude Code's session logs on disk, which
   may lag the live session by a few seconds. The 30s cadence is loose
   enough that this lag doesn't matter for the chart, but the report will
   note it in a footnote.

5. **One parent dir** — **`Naked-Claude-Experiments/` for all six**, with
   each subfolder name self-identifying via the `<timestamp>-<label>`
   convention (`2026-04-06-1255-NakedV1`, `2026-04-06-1310-EffortlessV2`,
   etc.). Runs are expected to happen back-to-back, sharing the parent
   directory.

All five resolved — the three scripts can now be written in one pass.
