# Trust Model

## Core principle

A mathematical conclusion is only as internal as its deepest load-bearing child. The system must report the trust boundary rather than hide it.

## Required distinctions

- **Parent edge removed:** a theorem-sized transition has been replaced by child obligations.
- **Parent fact derived:** the parent conclusion follows from those child obligations.
- **Theorem fully internalized for scope:** every load-bearing child is represented above the declared foundation and execution boundary.
- **Zero imports:** meaningful only relative to an explicitly named logic kernel, arithmetic substrate, and foundational axiom system.

These are separate facts and must never be represented by one boolean.

## Imported providers

The seven provider theorem stubs are currently imported. Their finite compiler tables are interfaces, tests, and counterfactual engines; they do not by themselves establish the universal theorem.

## Execution trust

The starter trusts:

1. JSON parsing and deterministic rulebook projection;
2. the configured execution substrates and their conformance harness;
3. the frozen v21 SQLite and archive as migration answer keys;
4. external mathematics explicitly recorded as provider imports.

## Failure behavior

Fail loudly on:

- missing source artifacts;
- hash mismatches;
- unknown theorem or provider IDs;
- status upgrades unsupported by child status;
- dependency cycles that are not explicitly modeled fixed points;
- a contradiction inserted as input rather than derived;
- conformance disagreement between projections.

No silent fallback to legacy paths or placeholder theorem statuses.
