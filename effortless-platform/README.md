# effortless-platform

This folder is the ERB tool eating its own dog food.

The repo at large manages a collection of demo rulebooks (`rulebook-examples/`). This folder is the wrapper that makes that possible — the admin portal, the build pipeline, the Postgres scripts, and the meta-rulebook that describes the tool itself all live here together as one self-contained Effortless project.

Everything outside this folder is a *document*. This folder is *the app that opens documents*.

---

## Sections

### `effortless-rulebook/`

The platform rulebook — `effortless-rulebook.json`.

This describes the ERB orchestration tool itself: its data model, navigation, APIs, permissions, build pipeline, and runtime configuration. It is the single source of truth for all platform-level concerns. Hand-edit it; run `effortless build` from this directory to regenerate derived artifacts (Postgres SQL, docs, etc.).

This file is categorically different from the per-domain rulebooks under `rulebook-examples/`. Do not mix them.

### `effortless.json`

The project manifest for the platform as an Effortless project. Declares the transpilers that run when you call `effortless build` here:

- **rulebooktopostgres** — generates the SQL in `postgres/` from the platform rulebook.
- **init-db** — runs `postgres/init-db.sh` to apply the generated SQL to `erb_admin_portal`.
- **JsonHbarsTransform** — generates `README.SCHEMA.md` at the repo root from the platform rulebook.
- **platform-rulebook-to-docs** — projects the platform rulebook into `docs/derived/` (one markdown file per table).

### `admin-portal/`

The ERB admin portal — an Express/Node.js backend (`server.js`) serving a small frontend (`web/`).

Runs on `localhost:7777`. Opens two Postgres connections simultaneously and for different reasons:

1. `erb_admin_portal` — the portal's own state (users, navigation, permissions). Read from the platform rulebook.
2. `erb_<domain>` — the document currently open. Determined by `orchestration/active-domain.txt`.

Write-through invariant: every mutating endpoint writes to both Postgres and the rulebook JSON in the same logical transaction. The rulebook JSON is always authoritative.

### `ssotme-proxy/`

A local HTTP server (`server.py`) on `localhost:4242` that makes injector scripts behave like first-class ssotme:// transpilers.

Each transpiler is a route:

```
POST http://localhost:4242/rulebook-to-python
POST http://localhost:4242/rulebook-to-postgres
POST http://localhost:4242/airtable-to-rulebook
```

The proxy sets `ERB_RULEBOOK_PATH` and `ERB_OUTPUT_DIR` and runs the corresponding injector script. No separate dispatch mechanism — the injector IS the route handler. Install into a demo project with:

```bash
effortless -install http://localhost:4242/rulebook-to-python \
    -i ../effortless-rulebook/<project>-rulebook.json
```

### `postgres/`

Generated SQL scripts that define the `erb_admin_portal` database schema.

Files are numbered to run in order (`00-bootstrap.sql` through `99-fk-constraints.sql`). Even-numbered files are generated by the `rulebooktopostgres` transpiler; odd `b`-suffixed files (e.g. `01b-customize-schema.sql`) are hand-editable overrides that are never overwritten by a rebuild. `init-db.sh` applies all scripts in sequence.

### `transpilers/`

Custom transpilers that are specific to this platform project and not general-purpose ssotme tools.

Currently contains `platform-rulebook-to-docs.py`, which reads the platform rulebook and projects it into `docs/derived/` as human-readable markdown. Generated files carry DO-NOT-EDIT headers and are removed by `effortless -clean`.

### `xlsx/`

Scratch area for Excel/XLSX output experiments. Not a full execution substrate — it does not participate in conformance testing.

### `execution-substrate-gt-explorer/`

A landing pad for exploring this repo from [Glamorous Toolkit](https://gtoolkit.com).

GT is used here as a code explorer and thinking surface, not as a graded conformance substrate. Contains Smalltalk playground snippets (`playground/*.st`) for inspecting the rulebook and generated artifacts live inside GT, plus starter notes for a Lepiter notebook. Nothing here is consumed by the orchestration pipeline.

---

## What does NOT live here

- Per-domain business tables (`Customers`, `Episodes`, `Tasks`, etc.) — those belong in `rulebook-examples/<domain>/`.
- The orchestration pipeline itself (`orchestrate.sh`, `shared.py`, formula parser, test runner) — that lives in `orchestration/`.
- The execution substrates for demo domains (Python, Go, Excel injectors) — those live in `execution-substrates/`.
