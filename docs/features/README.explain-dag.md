# ExplainDAG — Full Inference Tracing

> **For every derived value in your rulebook, ExplainDAG emits the complete
> derivation graph: which inputs were used, which operations were applied,
> and what the witnessed value was at each step.**

This is not a debugger or a log. It is a substrate — mechanically generated from
the same rulebook as Python, SQL, and Go — whose output is a witnessed,
reproducible, machine-readable proof that a given output follows from given
inputs under the declared rules.

---

## What it produces

ExplainDAG emits one JSON document per entity instance. Each calculated field in
the entity gets a node graph:

```json
{
  "entity": "Task",
  "instance": "employee-onboarding",
  "explanations": {
    "Name": {
      "nodes": {
        "i_result":  { "kind": "result",    "value": "employee-onboarding" },
        "i_op_1":    { "kind": "op",        "name": "SUBSTITUTE", "value": "employee-onboarding" },
        "i_ref_1":   { "kind": "field_ref", "field": "DisplayName", "value": "Employee Onboarding" }
      },
      "edges": [
        ["i_ref_1", "i_op_1"],
        ["i_op_1",  "i_result"]
      ]
    }
  }
}
```

Every node carries its witnessed value. Every edge is a directed dependency.
The full graph answers: "Why does this field equal this value?" — not by
re-running the computation, but by reading the proof.

---

## Why this matters

Most conformance tests tell you whether a substrate got the right answer.
ExplainDAG tells you *why* it got the answer — or, when something is wrong,
*where* the derivation diverged.

This has three practical consequences:

**1. Auditable business rules.** In regulated domains (finance, healthcare,
compliance), you often need to prove not just that a calculation was correct,
but *how* you arrived at it. ExplainDAG makes that proof a first-class artifact,
generated deterministically from the rulebook.

**2. Debugging without a debugger.** When a derived value is wrong, the DAG
shows exactly which upstream field carried the bad value. You do not need to
instrument code or add logging — the graph is already there.

**3. LLM-friendly reasoning.** The ExplainDAG JSON is the most LLM-readable
substrate in the catalog. A frontier model given an ExplainDAG document can
explain a calculation in plain English, identify which inputs to change to
produce a different output, and verify whether a change would propagate correctly.

---

## What "witnessed" means

A conventional DAG shows the *structure* of a computation: field A depends on
field B. ExplainDAG shows the *execution* of a computation: field A is
`"employee-onboarding"` because field B was `"Employee Onboarding"` and the
`SUBSTITUTE` operation replaced spaces with hyphens.

The values are not inferred — they are observed at generation time against the
rulebook's seed data. This is why ExplainDAG is a substrate: it runs the same
computation as Python or SQL, but its output is the derivation proof rather than
just the answer.

---

## Relationship to conformance

ExplainDAG participates in the conformance harness like any other substrate.
Its output format is deterministic: same rulebook + same seed data = same DAG.
If the DAG differs across builds, that is a bug in the transpiler or a change
in the rulebook — both are surfaced by the conformance report.

Because the DAG is deterministic, it can serve as an **answer key** for other
substrates. Instead of asking "does Python agree with Postgres?", you can ask
"does Python's output match the witnessed values in the ExplainDAG?" — which
gives you a substrate-neutral oracle.

---

## The DAG is not a log

Logs are written by running code. The ExplainDAG is generated by the rulebook
transpiler, before any production code runs. It describes what the rules *say*,
not what a particular implementation *did*. This distinction matters:

- A log can only exist after execution.
- An ExplainDAG can be generated from a rulebook that has never been deployed.

This means you can review the derivation proof for a proposed rule change before
shipping it, as part of the same build that generates your SQL migrations.
